Beachhead: Stretched-Cluster Split-Brain Recovery

First target use case for ASA — the pain everyone recognizes, the gap no one has closed

Autonomous Structural Arbitration (ASA) is a deterministic replica-arbitration layer for distributed storage systems, selecting a winning replica branch based on structural evidence rather than controller authority.

ASA sits above data protection (RAID, mirroring, erasure coding), below SDS / replication control planes. It deploys as an SDS plug-in or replication-layer module.

The problem

When a stretched cluster loses connectivity between sites, replicas diverge. When connectivity returns, the system must decide which branch of reality is correct. Today, that decision is made by controller authority, quorum votes, or manual intervention. All three have failure modes that cause data loss, extended outages, or operational confusion.

Wrong-branch selection

The controller or quorum picks the losing side. Writes committed on the winning branch are silently discarded or overwritten. Detection may take hours or days.

Brittle reconciliation

Resync workflows assume a clean winner exists. When both branches have valid, conflicting mutations, resync either drops data or requires manual intervention per-object.

Slow recovery debugging

When something goes wrong, operators must reconstruct what happened from controller logs, replication journals, and site-state snapshots — log archaeology under pressure.

How ASA resolves this

Partition occurs

Sites A and B lose connectivity. Both continue accepting writes independently. Replicas diverge.

ASA preserves branches

Instead of forcing a premature winner, ASA records structural metadata for each write on each site. Both branches are preserved as first-class objects with full causal lineage.

Connectivity returns

Sites exchange ASA metadata (not full data). Lineage hashes identify exactly where branches diverged and what each side did differently.

Scoring and decision

ASA scores each branch across four dimensions: causal coherence, structural consistency, event completeness, operator alignment. If the confidence gap exceeds Δ, the winner is selected automatically. If not, an arbitration ticket is surfaced to ops.

Reconciliation

The winning branch becomes canonical. ASA emits a state descriptor and schedules background repair. The losing branch is retained (with cause codes) for audit. No data is silently discarded.

Before and after

Without ASA

Controller picks a winner based on replication lag, timestamp, or quorum vote. Operator discovers the wrong branch was selected hours later during a consistency check. Recovery requires manual forensics across replication logs and site-state snapshots. Some data may be unrecoverable.

With ASA

Both branches are preserved with structural evidence. Scoring selects the winner deterministically. The losing branch is retained with full lineage for audit. Recovery debugging is structural inspection (compare score vectors, examine lineage hashes) — not log archaeology. Decision is auditable and reproducible.

Who feels this pain today

VMware / Broadcom

vSAN stretched clusters with witness-loss edge cases. Post-failure object availability depends on controller arbitration and quorum health. Deterministic site recovery would eliminate a class of escalations.

Nutanix / HCI

DR across sites where platform metadata and runbooks become hard to reason about under stress. State reconciliation across nodes and clusters is manual and error-prone.

Ceph / open-source storage

Cross-site RBD mirroring and RGW multi-site. Reconciliation depends on explicit resync behavior. PG scrubbing after partition is O(data), not O(divergence).

Enterprise arrays

Metro-cluster replication (NetApp MetroCluster, Dell SRDF, Pure ActiveCluster). Controller-based arbitration works but is opaque. ASA offers a software-native correctness differentiator that makes the arbitration decision auditable.

Why this beachhead

Already legible: Every storage team has a split-brain war story. The pain is understood across VMware, Nutanix, Ceph, and enterprise array audiences. No education required on why the problem matters.

Scoped: Split-brain recovery is a specific, bounded problem — not "solve all distributed consistency." ASA does not need to handle every edge case on day one; it needs to handle this one better than the current approach.

Measurable: Success criteria are concrete: correct branch selection rate, time-to-resolution, data preserved vs discarded, audit trail completeness. These are things you can benchmark.

Expandable: Once ASA proves value for split-brain recovery, the same primitive (branch preservation + structural scoring + deterministic selection) extends to other divergence scenarios: cross-region replication, snapshot consistency groups, multi-cluster state reconciliation.

Non-goals

ASA does not replace RAID, mirroring, erasure coding, or hardware controllers. ASA does not enforce consistency via locking or global quorum; it arbitrates between branches after divergence. ASA does not require application changes for the initial beachhead.
