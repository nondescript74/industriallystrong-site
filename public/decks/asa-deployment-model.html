ASA Deployment Model

Where ASA runs, how it integrates, and what it does not replace

Stack placement

Application / Database / Data Service
↓
VMware / Nutanix / Kubernetes / SDS control plane
↓
◆ ASA — deterministic replica-arbitration layer
↓
RAID / mirroring / erasure coding / JBOD / cloud block-object substrate

ASA sits above data protection, below distributed control-plane reconciliation.

Deployment models

SDS plug-in / replication-layer module Primary

ASA integrates as a module within the software-defined storage or replication layer. It intercepts replication events, records structural metadata inline, and exposes the arbitration API to the control plane above.

Best for: Ceph (RADOS plug-in), GlusterFS (translator), ZFS-on-Linux (channel program), VMware vSAN (provider), Nutanix AOS (stargate hook).

Why primary: Lowest latency insertion point. Feels like infrastructure, not an add-on. Aligns with the storage audience's mental model of where correctness belongs.

Node-local agent Secondary

For vendors with more fragmented stacks or heterogeneous storage substrates, ASA runs as a daemon on each storage node. It watches replication traffic via a shim or tap, maintains its own metadata log, and exposes the same arbitration API.

Best for: mdadm/JBOD environments, mixed-vendor metro clusters, legacy arrays with limited API surface.

Trade-off: Slightly higher latency (out-of-path observation), but broader compatibility.

Sidecar library Avoid leading with

A sidecar or library model is technically viable but sounds looser and less infrastructure-native for storage audiences. It implies ASA is optional middleware rather than a core correctness subsystem. Reserve this framing for application-layer integrations (e.g., database replication) where the storage substrate is opaque.

What ASA replaces (and does not replace)

ASA replaces

The arbitration decision — the moment when the system must choose which replica branch to trust after divergence. That decision becomes deterministic and auditable instead of controller-led or manual.

ASA does not replace

The storage stack. Not the control plane. Not replication transport. Not RAID. Not the filesystem. ASA is a correctness layer that composes with existing infrastructure, not a rip-and-replace story.

Integration surface

Write path

A single hook: record_event(object_key, event_metadata) called inline in the replication or SDS write path. Overhead is comparable to existing checksum or dedup header computation (32–64 bytes per block, ~1–3% metadata overhead).

Read/arbitration path

get_branches(), get_arbitration_ticket(), and resolve_ticket() are called only during divergence detection or recovery — not on the hot read path. Zero overhead on normal reads.

Telemetry

Divergence rate, auto-collapse rate, deferral rate, score-gap distribution, and failure-class breakdown (order non-determinism vs semantic ambiguity). Exportable to Prometheus, Datadog, or any metrics pipeline.

Operational posture

Failure mode: If ASA itself is unavailable, the underlying storage stack continues to operate normally — ASA is additive, not blocking. Divergences detected during ASA downtime are queued and scored on recovery.

Upgrade path: Metadata schema is versioned. New scoring dimensions or event fields can be added without reprocessing historical branches. Old branches retain their original score vectors.

Multi-site: ASA metadata replicates alongside the data it describes. Cross-site arbitration uses the same scoring engine; no separate coordination channel required.
