ASA Arbitration Primitive

One-page spec: how ASA records, scores, and resolves competing replica branches

1. Scope and insertion point

Layer: Above data protection (RAID, mirroring, erasure coding), below SDS / replication control planes.

Objects: Replica states for a given key (file, object, leg set, or logical record group).

Events: Mutations applied to those objects (open, amend, cancel, fill, rename, write, etc.).

2. Per-mutation metadata

For every mutation that reaches a replica, ASA records a compact event record:

  • event_id — globally unique logical event identifier.
  • replica_id — identity of the node / site that processed the event.
  • branch_id — identifier of the candidate branch this mutation extends.
  • parent_event_ids[] — one or more parent pointers for causal ancestry.
  • op_type — operation class (e.g., OPEN, AMEND, CANCEL, FILL, WRITE, DELETE).
  • position_map — structure/leg/extent indices the event touches (e.g., L1/L2/L3, byte ranges, key ranges).
  • completeness_flags — markers for required legs/fields present, missing, or partial.
  • intent_markers (optional) — operator / policy hints, derived from source system if available.
  • content_hash — cryptographic or strong non-cryptographic hash over the resulting candidate state delta.
  • lineage_hash — rolling hash chaining parent state to child state: H' = H(parent) ⊕ H(delta).

Records are appended to an ASA metadata log and periodically compacted into branch checkpoints.

3. Branch model

For each logical object, ASA maintains one or more branches:

A branch is an ordered sequence of event records with:

  • branch_id
  • head_event_id
  • head_lineage_hash
  • score_vector (see scoring below)
  • status ∈ {ACTIVE, WINNER, DEFERRED, DISCARDED}

Branches are created when replicas diverge (different arrival orders or interpretations) and are retained until either the confidence gap justifies auto-collapse, or an operator / higher layer explicitly resolves them.

4. Scoring dimensions

On each relevant change (new event, branch extension, or divergence detection), ASA computes a score vector per branch:

Cc
Causal coherence — Measures whether the event sequence is consistent with declared causal dependencies (no missing parents, no impossible re-ordering). Penalizes gaps, cycles, or out-of-order application that violate parent_event_ids.
Sc
Structural consistency — Measures whether the resulting structure is internally valid: leg/extent counts, invariants, constraints, hedge ratios, etc. Penalizes invalid combinations (e.g., dangling legs, violated invariants).
Ec
Event completeness — Measures whether all required events for a known pattern have been seen (e.g., open+fill+close, or open+cancel). Penalizes branches with missing mandatory steps relative to peers.
Oa
Operator alignment — Measures alignment with operator / policy intent where available (intent markers, desk rules, configuration). Penalizes branches that contradict explicit intent or policy.

Each dimension is normalized to [0, 1]. Branch score is:

S = wcCc + wsSc + weEc + woOa

with weights wc, ws, we, wo chosen per deployment profile (e.g., safety-biased vs latency-biased).

5. Decision rules

Given two or more ACTIVE branches for the same object:

Auto-collapse

If one branch's score exceeds all others by at least a configured confidence gap Δ (e.g., Smax − Snext ≥ 0.2) and no hard constraint is violated:

  • Mark that branch WINNER
  • Mark competitors DISCARDED with cause codes
  • Emit audit record: winning branch_id, score vector, competing scores, key discriminating factors

Defer to arbitration

If scores are within Δ or any dimension signals unresolved ambiguity (incomplete event set, conflicting intent):

  • Keep competing branches ACTIVE
  • Expose arbitration ticket to higher layers / ops UI
  • Optionally accept operator decision or wait for more events

Preserve under concurrency

When concurrent mutations arrive (conflicting updates across sites):

  • Spawn or extend branches without auto-collapse
  • Re-score only when causal relationships or structural differences become clear

Reconciliation after winner selection

Once a branch is WINNER, ASA emits a canonical state descriptor (final lineage hash, structural snapshot) and schedules background repair so lagging replicas reconcile to the winner.

6. Interfaces

Write-side hook

record_event(object_key, event_metadata) — called inline in the replication/SDS path.

Arbitration API

get_branches(object_key) → [branch]

get_arbitration_ticket(object_key) → ticket?

resolve_ticket(object_key, winning_branch_id, reason)

Telemetry

Rate of divergences, auto-collapses, deferrals. Distribution of score gaps and failure classes (order non-determinism vs semantic ambiguity).

This is enough detail that a storage vendor or distributed systems engineer can see what ASA records, how it scores branches, and when it chooses to auto-collapse versus escalate, without forcing a particular implementation of hash type, weights, or storage layout.
